Python 개발자라면 한 번쯤 들어봤을 단어, **GIL(Global Interpreter Lock)**.

누군가는 "Python이 느린 이유"라고 하고, 누군가는 "Python이 쉬운 이유"라고도 합니다. 도대체 GIL이 무엇이길래 수십 년간 Python을 지배해왔으며, 2025년 말 현재 **'GIL 없는 Free-threading'**은 어디까지 왔을까요?

이 글에서는 GIL의 존재 이유와 역사, 그리고 최신 버전의 Python과 함께 도래하고 있는 새 시대를 탐구해 봅니다.

---

## 1. GIL, 넌 누구냐?

**GIL(Global Interpreter Lock)**은 CPython 인터프리터(우리가 쓰는 표준 Python)가 한 번에 하나의 스레드(Thread)만 Python 바이트코드를 실행하도록 강제하는 **뮤텍스(Mutex, 잠금 장치)**입니다.

쉽게 비유하자면, 노래방에서 **'단 하나의 마이크'**입니다.

* 노래방(프로세스) 안에 사람(스레드)이 4명, 8명이 있어도 상관없습니다.
* 마이크(GIL)가 딱 하나뿐이라서, **노래를 부를 수 있는 사람은 동시간대에 오직 1명**뿐입니다.
* 나머지 사람들은 마이크를 넘겨받길 기다려야 합니다.

그래서 Python에서 스레드를 여러 개 만들어도, CPU 코어를 다 활용하지 못하고 하나의 코어에서만 바쁘게 교체(Context Switching)되는 현상이 발생합니다.

---

## 2. 왜 이런 '족쇄'를 만들었을까? (The Rationale)

'귀도 반 로섬(Guido van Rossum)'은 왜 멀티 코어 시대에 역행하는 결정을 했을까요? 여기에는 Python의 태생적 설계 철학과 관련된 **두 가지 결정적인 이유**가 있습니다.

### 2-1. 메모리 관리의 단순화 (Reference Counting)

Python은 모든 객체의 '참조 횟수(Reference Count)'를 기록하여 메모리를 관리합니다.

```python
import sys
a = []
b = a
sys.getrefcount(a)  # 참조 카운트 확인
```

만약 GIL 없이 여러 스레드가 동시에 이 숫자를 건드리면(Race Condition), 메모리 누수(Leak)가 발생하거나 프로그램이 붕괴(Crash)될 위험이 큽니다. GIL은 이 복잡한 동기화 문제를 '락(Lock) 하나'로 해결하여, Python 내부 구현을 단순하고 안전하게 만들었습니다.

### 2-2. C 라이브러리와의 쉬운 결합 (Glue Language)

사실 더 현실적인 이유는 **Python을 'Glue Language'로 성공시키기 위해서**였습니다.
Python의 강력함은 `NumPy`, `SciPy` 등 수많은 고성능 C/C++ 라이브러리를 쉽게 가져다 쓸 수 있다는 점에 있습니다.

* **GIL이 없으면:** 복잡한 스레드 락(Lock)을 C 확장 모듈 개발자가 직접 구현해야 합니다. 이는 라이브러리 개발 난이도를 크게 높이므로 커다란 진입 장벽이 될 수 있습니다.
* **GIL이 있으면:** Python 인터프리터가 알아서 줄을 세워주므로, C 모듈 개발자는 로직에만 집중할 수 있습니다.

이 전략 덕분에 Python은 초기 생태계를 폭발적으로 확장할 수 있었지만, 역설적으로 지금에 와서는 Python의 발전을 가로막고 있는 **'기술 부채(Technical Debt)'**가 되었습니다.

## 3. GIL이 미치는 영향

GIL이 항상 나쁜 것만은 아닙니다. 작업의 종류에 따라 영향이 다릅니다.

### 🙂 I/O Bound 작업 (대기 중심)

* 파일 읽기/쓰기, 네트워크 요청(API) 작업
* 스레드가 I/O 작업(대기)에 들어가면 Python은 **"너 노래 안 부르지? 마이크 내놔"** 하고 GIL을 뺏어서 다른 스레드에게 줍니다.
* Python의 멀티 스레딩은 웹 크롤링이나 API 서버 등의 I/O 작업에서는 충분히 효율적입니다.

### 😐 CPU Bound 작업 (연산 중심)

* 수학 연산, 머신러닝 학습, 이미지 처리 등의 작업
* 스레드를 늘려도 마이크 쟁탈전(Lock 경쟁) 때문에 **오히려 싱글 스레드보다도 느려질 수 있습니다.**
* **과거의 해결책:** `multiprocessing`을 쓰거나, GIL을 일시 해제하는 C 확장 모듈을 사용해야 했습니다.

## 4. 격변의 시대: 실패의 역사와 PEP 703

GIL을 없애려는 시도는 이번이 처음이 아닙니다. 지난 20여 년간 수많은 개발자들이 이 문제에 도전했으나 번번이 좌절했습니다.

### 4-1. GIL 제거 시도의 역사 (Timeline)

가장 큰 난관은 **'싱글 스레드 성능 저하'**였습니다. GIL을 없애려면 Lock을 객체마다 걸어야 하는데, 이 오버헤드 때문에 스레드를 하나만 쓸 때 오히려 속도가 크게 느려지는 현상이 발생해 거절당하곤 했습니다.

| 시기 | 프로젝트명 (주도자) | 결과 | 비고 |
| :---: | :---: | :---: | :---: |
| **1999** | **Free-Threading Patch**<br>(Greg Stein) | **실패** | 싱글 스레드 속도 40% 저하로 도입 무산 |
| **2016** | **The Gilectomy**<br>(Larry Hastings) | **중단** | 유지보수 복잡도 증가 및 성능 문제 해결 실패 |
| **2021** | **nogil**<br>(Sam Gross) | **기반 마련** | 'Biased Reference Counting' 등으로 성능 저하 최소화<br>(PEP 703의 기반이 된 프로토타입) |
| **2023** | **PEP 703 승인** | **채택** | Python 3.13부터 단계적 도입 확정 |
| **2025** | **Python 3.14** | **배포** | Free-threading 모드 공식 지원<br>안정화 진행 단계 |
{: style="width: 100%; border: 2px solid #DCE0E2;"}

### 4-2. 이번엔 무엇이 다른가? (PEP 703)

2021년, Sam Gross가 제안한 `nogil` 프로젝트는 기존의 실패 요인을 기술적으로 극복했습니다.

* **Biased Reference Counting(편향된 참조 카운팅)**
    * 기존에는 모든 스레드가 공유 객체의 카운트를 수정하려 Lock을 걸었습니다.
    * `nogil`은 해당 객체를 만든 '주인 스레드'는 Lock 없이 빠르게 카운트를 수정하고, '다른 스레드'가 접근할 때만 Lock을 거는 방식을 사용해 오버헤드를 획기적으로 줄였습니다.

* **Immortal Objects(불멸의 객체)**
    * `None`, `True` 같은 기본 객체는 참조 카운트 연산을 아예 생략하는 등 불필요한 연산을 제거했습니다.

### 4-3. 실제 성능 향상

Python 3.14의 Free-threading 모드(`--disable-gil` 빌드)를 사용했을 때 얻을 수 있는 이점은 명확합니다.

* **멀티 코어 확장성(Scalability)**
    * **과거:** 코어가 아무리 많아도 Python 연산 속도는 1개일 때와 동일합니다. (1x)
    * **현재:** 작업 특성에 따라 다르지만 일반적으로 코어 수에 비례하여 성능이 향상됩니다. 예를 들어, CPU 집약적 연산에서 코어를 20개 사용 시 이론적으로 10~20배의 속도 향상이 가능합니다. 단, 벤치마크에 따라 상이할 수 있으며 3~10배 정도가 일반적입니다. (e.g., Fibonacci 3.1x, Prime 10x)

* **어디에 쓸 수 있을까?**
    * 순수 Python으로 작성된 웹 서버 프레임워크의 처리량(Throughput) 대폭 향상
    * 데이터 전처리, 암호화/복호화, 이미지 처리, 과학 계산 등 CPU Bound 작업을 별도의 C 모듈 없이 순수 Python 코드만으로 병렬 처리 가능

> **싱글 스레드 성능(Single-thread Performance)**
>
> GIL을 없애는 과정에서 약간의 오버헤드는 피할 수 없습니다. Python 3.14 기준, 싱글 스레드 성능에서 일반 모드 대비 약 **5~10% 정도의 속도 저하**가 발생할 수 있습니다. 하지만 멀티 코어를 활용했을 때 얻는 이득이 이 손해를 압도하므로, 병렬 처리가 필요한 환경에서는 매우 합리적입니다.

## 맺음말: 우리는 무엇을 준비해야 할까?

2025년이 끝나가는 지금, Python은 거대한 과도기에 있습니다.

1. **일반 개발자:** 당장 코드를 바꿀 필요는 없습니다. 하지만 최신의 Python 환경에서는 멀티 스레딩을 활용한 CPU 병렬 처리가 가능해졌다는 점을 인지하고, 성능 테스트를 해볼 시점입니다.
2. **라이브러리 관리자:** GIL 덕분에 신경 쓰지 않았던 스레드 안정성(Thread Safety)을 이제는 직접 챙겨야 합니다.

### 요약

* GIL은 **안전한 메모리 관리**와 **쉬운 C 라이브러리 연동**을 위한 전략적 선택이었습니다.
* 이 덕분에 Python 생태계가 성장했지만, 멀티 코어 성능에 제한이 있었습니다.
* Python 3.14(2025)에서 선택적 지원으로 **GIL 없는 병렬 처리의 시대**가 열렸고, 현재 개발 중인 3.15(2026) 이후의 버전부터는 Free-threaded 빌드의 기본화 논의가 본격화될 전망입니다. 어쩌면 우리는 향후 몇 년 안에 GIL이라는 단어를 역사책에서나 보게 될지도 모릅니다.
