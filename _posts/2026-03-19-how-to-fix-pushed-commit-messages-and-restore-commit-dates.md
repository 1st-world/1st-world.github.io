예전에 진행했던 프로젝트의 커밋 기록들을 쭉 훑어보다가, 과거에 썼던 커밋 메시지들이 요즘 사용하는 현대적인 컨벤션(형식)에 맞지 않아 영 거슬리더라고요. 그래서 깔끔하게 다듬어보자는 마음에 메시지를 수정했습니다. 

그런데, GitHub에 들어가 보니 잔디밭과 타임라인의 흐름이 엉망이 되어버렸습니다! 😱 해당 커밋의 시점이 '수정한 현재 시각'으로 덮어씌워졌기 때문이었어요. 프로젝트 흐름이 꼬여버려서 꽤나 당황했었죠.

그래서 오늘은 커밋 메시지를 수정하는 기본적인 방법부터, 커밋 시점을 유지하며 수정하는 방법, 원격 저장소에 푸시된 내역을 안전하게 덮어쓰기, 그리고 이미 엎질러진 물('현재 시각'으로 변경된 채 푸시한 상황)을 `reflog`를 활용해 복원하는 과정까지 차근차근 정리해 보겠습니다.

---

## 1. 커밋 메시지 수정하기

커밋 메시지를 수정하는 방법은 수정하려는 커밋의 위치에 따라 두 가지로 나뉩니다.

### 1-1. 최근 커밋 메시지 수정 (`--amend`)

직전에 남긴 최근의 커밋 메시지를 수정하려면 `--amend` 옵션을 사용합니다.

```bash
git commit --amend
```

이 명령어를 입력하면 기본으로 설정된 에디터(Vim, Nano 등)가 열릴 거예요. 보통 첫 줄에는 커밋 제목을 작성하고, 한 줄을 띄운 뒤 세 번째 줄부터 상세한 본문을 작성하여 여러 줄의 커밋 메시지를 예쁘게 구성하면 됩니다. 수정을 완료하고 저장 후 에디터를 닫으면 됩니다.

### 1-2. 과거의 특정 커밋 메시지 수정 (`rebase -i`)

최근이 아닌, 예전에 남겼던 여러 커밋 중 특정 커밋의 메시지만 골라서 수정하고 싶다면 **인터랙티브 리베이스(Interactive Rebase)**를 사용해야 합니다. 

예를 들어, 최근 3개의 커밋 목록을 펼쳐보고 싶다면 아래 명령어를 입력합니다. (만약 과거의 특정 커밋부터 보고 싶은 경우 `git rebase -i [해시값]^` 형태로 입력하면 됩니다.)

```bash
git rebase -i HEAD~3
```

그러면 에디터가 열리면서 커밋 목록이 **오래된 순서대로(위에서부터 아래로)** 나타납니다.

```text
pick 3a2b1c0 첫 번째 커밋 메시지
pick 9f8e7d6 두 번째 커밋 메시지 (수정하고 싶은 커밋!)
pick 5a4b3c2 세 번째 커밋 메시지
```

여기서 수정하고 싶은 커밋 맨 앞의 단어 `pick`을 **`reword`**(또는 짧게 `r`)로 변경해 줍니다.

```text
pick 3a2b1c0 첫 번째 커밋 메시지
reword 9f8e7d6 두 번째 커밋 메시지 (수정하고 싶은 커밋!)
pick 5a4b3c2 세 번째 커밋 메시지
```

이 상태로 저장하고 에디터를 닫으면 `reword`로 지정했던 커밋의 메시지를 수정할 수 있는 새로운 에디터 창이 다시 열립니다. 여기서 앞서 설명한 것처럼 원하는 대로 메시지를 수정한 뒤 저장하고 닫으면 완료됩니다!

이렇게 과거의 커밋을 수정하게 되면 해당 커밋은 물론이고 **그 후에 이어져 있던 모든 커밋들의 해시(Hash) 값도 새롭게 변경**되는데요. 따라서 이미 원격 저장소에 올라간 상태라면 반드시 '강제 푸시' 과정이 필요한데, 이건 잠시 후에 이어서 설명 드리겠습니다.

## 2. 커밋 시점(Date)을 유지하면서 수정하기

Git의 커밋 시점 정보에는 코드를 처음 작성한 **Author Date**와 커밋이 마지막으로 수정된 **Committer Date**, 이렇게 두 가지가 있습니다.

앞선 방법으로 단순히 메시지를 수정하면, 코드를 처음 작성한 날짜(Author Date)는 그대로인 반면 **커밋이 마지막으로 수정된 날짜(Committer Date)가 '현재 시각'으로 변경**되어 버립니다. 커밋을 다시 하는 것이니 수정 일시가 바뀌는 건 자연스러운 일이지만, 이로 인해 의도치 않게 타임라인이 꼬이는 등의 문제가 생길 수도 있어요.

그래서 이럴 때는 다음과 같은 방법들을 활용해 시점을 잡아두는 것을 추천합니다.

### 2-1. 최근 커밋 수정 (`--amend`)

현재 HEAD가 가리키는 기존 커밋의 시간을 그대로 가져와서 Committer Date에 덮어씌우며 수정합니다.

```bash
GIT_COMMITTER_DATE="$(git show -s --format=%ci HEAD)" git commit --amend
```

### 2-2. 과거의 특정 커밋 수정 (`rebase -i`)

리베이스를 할 때는 `--committer-date-is-author-date` 옵션 하나만 추가해 주면 간단하게 Committer Date를 원래의 Author Date와 동일하게 유지할 수 있습니다.

```bash
git rebase -i --committer-date-is-author-date HEAD~3
```

이러한 방법들을 쓰면 커밋 메시지만 깔끔하게 바꾸고 날짜는 과거의 제자리에 남겨둘 수 있습니다!

## 3. 안전한 강제 푸시

로컬에서 커밋 메시지를 수정했다면, Git은 이를 완전히 새로운 커밋으로 인식하여 해시(Hash) 값을 변경합니다. 따라서 이미 원격 저장소(GitHub 등)에 올라가 있는 상태라면 일반적인 `git push`만으로는 반영할 수 없고 '강제 푸시' 옵션을 부여해야 합니다.

이때 `--force` 옵션 대신 **`--force-with-lease` 사용을 강력히 권장**합니다.

* **`--force` (위험):** 나의 로컬 상태로 원격 저장소를 덮어씁니다. 만약 내가 모르는 사이 다른 팀원이 원격 저장소에 새로운 코드를 푸시해 두었다면, 그 코드가 통째로 날아가 버리는 참사가 벌어질 수 있습니다.

* **`--force-with-lease` (안전):** 내가 알고 있는 원격 저장소의 상태와 실제 상태가 일치할 때만 푸시를 진행합니다. 즉, 내가 풀(Pull)을 한 뒤에 누군가 새로운 커밋을 추가했다면 푸시를 막아 새 코드를 보호합니다.

```bash
git push --force-with-lease
```

혼자 진행하는 개인 프로젝트라면 `--force`를 사용해도 무방하지만, 추후에 협업 환경에서 실수하는 것을 방지하려면 평소에 `--force-with-lease`로 사용하는 습관을 들이는 것이 좋습니다.

## 4. 이미 엎질러진 시간 복원하기

혹시 메시지만 예쁘게 바꾸려 했는데 깜빡해서 커밋 시점을 보존하지 못한 채로 푸시해버린 난감한 상황에 처하셨나요? 제가 겪었던 일인데요… 😭 이때 Git의 기능 중 `reflog`를 활용해 과거를 되돌려 보겠습니다.

### 4-1. 잃어버린 커밋 해시 찾기

`git reflog` 명령어는 브랜치의 HEAD(현재 위치)가 가리켰던 모든 변경 기록을 순서대로 보여줍니다.

```bash
git reflog
```

입력하면 아래와 비슷한 형태의 목록이 출력됩니다.

```text
a1b2c3d (HEAD -> main) HEAD@{0}: commit (amend): 수정한 커밋 메시지 (현재 꼬여버린 커밋)
e4f5g6h HEAD@{1}: commit: 수정하기 전의 커밋 메시지 (우리가 찾던 과거의 커밋)
...
```

여기서 가장 위에 있는 `HEAD@{0}`은 방금 수정한 커밋입니다. 그 바로 아래에 있는 `HEAD@{1}`의 해시 값 `e4f5g6h`이 바로 **수정하기 전의 원래 시점을 가지고 있는 커밋 해시**입니다.

### 4-2. 원래 시점 확인하기

찾아낸 예전 커밋 해시가 품고 있던 원래 날짜와 시각을 확인해 볼까요? `git show` 명령어에 `--format=fuller` 옵션을 함께 주면 상세 정보를 볼 수 있습니다.

```bash
git show -s --format=fuller e4f5g6h
```

입력하면 아래와 비슷한 형태의 목록이 출력됩니다.

```text
commit e4f5g6h...
Author: MyName <my.email@example.com>
AuthorDate: Wed Oct 15 14:00:00 2025 +0900
Commit: MyName <my.email@example.com>
CommitDate: Wed Oct 15 14:00:00 2025 +0900
```

여기서 `AuthorDate`와 `CommitDate`가 우리가 복원해야 할 기존 시점 정보입니다.

### 4-3. 이전 커밋의 시점 복원 적용하기

이제 찾아낸 해시 값을 이용해 현재 커밋의 날짜를 기존 시점으로 고정해 줍니다. 

```bash
# 원래 커밋 해시를 변수에 저장 (본인이 찾은 해시 값 입력)
OLD_HASH="e4f5g6h"

# 원래 커밋의 정확한 날짜 포맷 추출
OLD_DATE=$(git show -s --format=%ci $OLD_HASH)

# Committer Date, Author Date를 모두 원래 시간으로 덮어쓰며 수정 적용
GIT_COMMITTER_DATE="$OLD_DATE" git commit --amend --no-edit --date="$OLD_DATE"
```

참고로 `--no-edit` 옵션은 이미 수정한 커밋 메시지 본문을 다시 편집하지 않고 그대로 유지하겠다는 의미입니다.

### 4-4. 다시 강제 푸시

시점 정보가 복원되었는지 `git log --format=fuller`로 확인한 후, 원격 저장소에 다시 강제 푸시합니다.

```bash
git push --force-with-lease
```

이렇게 해서, 커밋 메시지를 수정하고 타임라인과 날짜는 과거의 올바른 위치로 무사히 되돌렸습니다!
